=====================================================
웨이퍼맵 결함 분류 시스템 - MVP 로드맵 (4조) v4
(v3 대비: AI 트랙을 실제 데이터셋/모델 기준으로 전면 수정)
=====================================================

[기술 스택 결정사항]
-----------------------------------------------------
* 아키텍처 확정: 옵션 B (백엔드 서버 / AI 서버 분리, 둘 다 FastAPI)
  - "FastAPI 하나가 업로드+추론+DB저장 다 처리" 방식은 채택하지 않음
  - AI팀은 AI 서버만 담당, "이미지 넣으면 결과 반환" API 계약만 지키면 됨
  - 백엔드팀은 AI 서버 내부 코드(모델 로딩 등)를 알 필요 없음 → 팀 간 경계 명확
* 백엔드: Python / FastAPI 단일 스택 (Spring Boot 미채택)
* API 필드 표기: snake_case (예: analysis_id, class_name)
* 저장소: MySQL 단일 사용 (블록체인 미채택)
* 서버 구성 (프로세스 3개, 로컬 개발 시 각각 실행 필요)
  - 백엔드 서버 (FastAPI): 업로드 처리, 이력/통계 조회, DB 저장 — 포트 8000
  - AI 서버 (FastAPI): 모델 로딩, 이미지 전처리, 추론 전용 — 포트 8001
  - MySQL: 별도 DB 서버 — 포트 3306
  - 3개를 매번 따로 켜는 게 번거로우면 Docker Compose로 묶어서
    `docker-compose up` 한 번에 실행하는 걸 권장

* 모델 서빙 방식 확정: 학습은 오프라인 1회 (baseline), 추론 시 재학습 없음
  - 학습된 모델 파일은 AI 서버 기동(startup) 시 1회만 메모리에 로드
  - 요청(이미지 업로드)마다 모델을 디스크에서 다시 로드하지 않음
  - FastAPI 구현 시 `@app.on_event("startup")` 또는 전역 변수에 모델 객체 유지
  - 업로드 시 발생하는 연산은 추론(inference)뿐이며, 학습(training)은 발생하지 않음

  [!! 실제 모델 파일 구성 - v4에서 확정 !!]
  AI 서버 startup 시 로드해야 하는 파일은 2개이며, 반드시 같은 폴더에 있어야 함:
    · resnet9_wafer.pth   : 모델 가중치 (torch.load + load_state_dict로 로드)
    · label_mapping.json  : 클래스 이름 <-> class_id 매핑
                            (예: {"Center":0, "Donut":1, ..., "none":8})
  .pth 파일에는 가중치 숫자만 들어있고 "0번이 어느 클래스인지"는 안 들어있음.
  label_mapping.json 없이 .pth만 배포하면 추론 결과가 조용히 틀린 클래스명으로
  나갈 위험이 있음 — 두 파일 항상 세트로 취급할 것.

  [!! 용어 구분 - 헷갈리기 쉬운 지점 !!]
  "서버 시작 시 모델 로딩"과 "모델 학습"은 완전히 다른 연산임:
    · 학습(training)  : 1-B 단계에서 오프라인으로 1회만 수행. 아래 [AI 데이터셋]
                         기준 약 17만 장으로 gradient 계산/weight 업데이트 반복.
                         GPU 기준 수십 분(20~40 epoch), CPU면 수 시간 이상 소요될 수 있음.
                         결과물을 resnet9_wafer.pth + label_mapping.json으로 저장.
    · 로딩(loading)   : AI 서버가 켜질 때마다 실행. 이미 저장된 파일들을
                         메모리에 올리기만 함. 1초 미만 소요.
  "서버 켤 때마다 오래 걸리는" 상황이 발생한다면, startup 코드 안에 학습 루프가
  잘못 들어가 있는 것 — 구현 시 반드시 확인.

[역할 분담]
-----------------------------------------------------
프론트     : 업로드, 미리보기, 결과, 이력, 통계, 오류 처리 (React)
백엔드1(추서림)   : 이미지 수신 API — 업로드 엔드포인트, AI 서버 호출, 응답 변환, 장애 처리
백엔드2(성광) : 결과 저장/조회 API — 분석 이력 목록, 상세 조회, 통계 집계
백엔드3(최규라) : MySQL DB 스키마 설계, 쿼리 최적화, 백엔드1·2와 인터페이스 협의
AI         : CNN 학습 (ResNet9 기반, 아래 [AI 데이터셋/모델 확정 사항] 참고)
AI 서버    : 모델 로딩(서버 기동 시 1회), 이미지 전처리, 추론 API (FastAPI)

* 백엔드1·2·3은 하나의 FastAPI 프로젝트 안에서 라우터(router)/모듈 단위로 분리
* 전담 QA 역할 없음 — Phase 3 통합 테스트는 팀 전체가 각자 파트 기준으로 분담

[사전 확정 - Phase 0]
-----------------------------------------------------
0-1. 클래스 구조 확정: 9 classes (정상 1 + 결함 8) - 이미 EDA로 확정
     (none, Center, Donut, Edge-Ring, Edge-Loc, Loc, Random, Scratch, Near-full)
0-2. API 응답 규격 확정 (프론트·백엔드·AI 공동, snake_case 기준)
     예: { class_id, class_name, confidence, image_id }
     -> 이 규격은 이후 각 트랙이 목업으로 병렬 개발하는 기준선
     -> 이 응답은 추론 결과이며, 학습 데이터로 재사용되지 않음 (별도 파이프라인 없음)

  [!! 업로드 이미지 포맷 - v4에서 새로 확정 (중요) !!]
  학습 데이터(waferMap)는 사진이 아니라 다이(die) 단위 categorical 값
  (0=웨이퍼 밖, 1=정상, 2=불량) 배열임. 프론트에서 받는 업로드 이미지가
  이 값을 그대로 보존하는 포맷이어야 추론이 정상 작동함:
    · PNG 팔레트 모드(mode='P')로, 픽셀 인덱스가 0/1/2 그대로 유지되어야 함
    · 업로드/저장/전송 전 구간에서 무손실 유지 (JPEG 변환 절대 금지)
    · 리사이즈가 필요한 경우 반드시 최근접 보간(nearest neighbor) 사용
      (bilinear/bicubic 쓰면 0/1/2 사이 중간값이 생겨서 categorical 의미가 깨짐)
  프론트/백엔드가 일반적인 "이미지 업로드"처럼 썸네일 생성이나 JPEG 압축을
  무심코 넣으면 이 값이 깨져서 AI 서버가 엉뚱한 예측을 하게 됨 — 반드시 공유.

[AI 데이터셋/모델 확정 사항 - v4 신규]
-----------------------------------------------------
* 데이터셋: WM-811K (LSWMD.pkl), 라벨 있는 데이터 약 172,950장
  (902장 서브셋 방식은 v3까지의 임시 계획이었고, 실제로는 전체 라벨 데이터 사용으로 변경)
* 데이터 분할: 학습 90% / real held-out test 10% (pkl 자체 분할, stratify 적용)
  + 별도로 클래스당 100장씩(Near-full은 보유량 부족으로 49장) 완전히 학습에서
    제외한 진짜 held-out 검증셋(real_holdout_100) 추가 확보 — 데이터 누수 없음을
    인덱스 대응 검증까지 거쳐서 확인함
* 클래스 불균형 처리: none 10,000장 / 나머지 결함 클래스 5,000장 목표로
  오버샘플링(+flip/rotate/translate 증강), Near-full은 원본 부족으로 증강 비율 상한 적용
* 이미지 크기: 64x64 (32x32 대비 실측 비교 후 확정 — 32는 명확히 성능 하락)
* 모델: ResNet9 (fast.ai 스타일, from scratch) + 256채널 단계 residual block 추가
* Class weight: none=2.3, Near-full=1.5, 나머지=1.0 (여러 차례 실측 튜닝 후 확정)
* 정규화: Dropout(0.4) + Dropout2d(0.2) + weight_decay(1e-4) + Early Stopping
* 실측 성능 (real_holdout_100, 클래스 균형 데이터 기준):
  - 전체 정확도(micro): 92.2%
  - Macro F1: 0.92
  - 대규모 pkl held-out(3만 4천여 개, 실제 클래스 비율 그대로) 기준: 정확도 96%대, Macro F1 0.85~0.86
* 알려진 약점: Loc/Edge-Loc/none 경계에서 상대적으로 혼동 발생
  (균형 잡힌 데이터에서는 이 격차가 완화됨을 확인 — 불균형 테스트의 통계적 불안정성 때문일 가능성 있음)
* 합성(synth) 데이터 실험 결과: 학습 데이터에 합성 이미지를 섞는 실험(소량/집중 주입 둘 다)을
  진행했으나 실제 성능 개선 없음(오히려 소폭 하락) — 최종적으로 실제 데이터(REAL)만
  학습에 사용하는 것으로 확정. 합성 데이터는 파이프라인 동작 확인(sanity check) 용도로만 활용.
